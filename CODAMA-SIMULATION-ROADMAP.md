## Executive Summary

This roadmap merges three work streams into one sequenced plan:

1. **@solana/kit migration** — Replace `@solana/web3.js` v1 + `@coral-xyz/anchor` TS layer with Kit-native code generated by Codama. The on-chain Anchor program (Rust) is unchanged. Only the TypeScript client layer changes.
2. **Codama IDL pipeline** — Dynamic protocol onboarding via Anchor IDL → Kit-native instruction builders. Proven: 8/8 protocols, 100% instruction coverage, 0.3ms/instruction.
3. **Transaction simulation safety net** — Pre-sign balance-delta analysis with drain detection. Proven: 27/27 stress tests, 3 flag types, Helius enhanced fields.

**Core insight**: The frontend does NOT use Anchor 0.32.1. The on-chain program uses Anchor (Rust). The TypeScript SDK uses `@coral-xyz/anchor` and `@solana/web3.js` — but it doesn't have to. Codama generates a complete Kit-native TypeScript client from the Anchor IDL. This means we can go full `@solana/kit` on the client side with zero Anchor TypeScript dependency.

**Strategy**: Orca Whirlpools pattern — build a NEW `sdk/kit/` package (`@phalnx/kit`) alongside the existing `sdk/typescript/` (`@phalnx/sdk`). Both coexist. No changes to the existing SDK. Downstream packages migrate at their own pace. Old SDK enters maintenance mode after new one is production-ready.

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Migration pattern | Parallel SDK (Orca pattern) | No risk to existing consumers. Codama generates 90% of code. |
| Kit version | Pin `@solana/kit@6.2.0` exact | Kit has 6 major versions in 15 months. Pin for stability. |
| Anchor TS removal | Complete — via Codama | Codama replaces `Program<Phalnx>`, `BN`, `PublicKey` entirely. |
| T2 protocol SDKs | Compat bridge internally | drift-ts, flash-sdk, klend-sdk still use web3.js. Bridge at boundary. |
| T3 onboarding | Codama IDL → Kit-native handlers | 100% proven in stress tests. O(n) linear scaling. |
| Actions/Blinks | DROP | Jupiter-only in practice. Already covered by T1_API. |

**Benefits of Kit-native SDK:**
- 78% smaller bundle (tree-shakeable, zero dependencies)
- 10x faster Ed25519 crypto (native APIs vs JS polyfills)
- Full TypeScript type safety (`Address` branded strings, native `bigint`)
- Zero `@coral-xyz/anchor` supply chain risk in client code
- Codama-generated code is first-class (no adapter needed)
- Functionally identical — same instructions, same accounts, same on-chain program

### 2.4 What's NOT Wired Together (The Problem)

Only 5/31 intent types execute end-to-end — 84% of actions fail. The pieces exist but are NOT connected:
- `IntentEngine.execute()` does not call `ProtocolHandler.compose()`
- `resolveAccounts()` is not called in the compose pipeline
- `precheck()` does not compare intent amount vs spending cap
- `toAgentError()` is not used in all error paths
- `T3_IDL` tier exists in the enum but is never assigned
- `T4_PASSTHROUGH` has no constraints enforcement
- Simulation runs post-compose only, not pre-sign

### Phase 0: Security Fixes in Current SDK (P0 — 1-2 days)

Fix 5 critical audit findings that produce incorrect behavior in `sdk/typescript/`. These fixes ship immediately and also inform the new SDK design (sdk/kit/ will not have these bugs by construction).

#### Step 0.1: H-1 — Fix precheck permission mapping

- **File**: `sdk/typescript/src/client.ts` (~line 1791)

  ```typescript
  const EVENT_DISCRIMINATOR_SIZE = 8; // SHA256("event:<EventName>")[..8]

  export interface ParsedEvent {
    name: string;
    data: Record<string, unknown>;
  }

  export function parsePhalnxEvents(logs: string[]): ParsedEvent[] {
    const events: ParsedEvent[] = [];
    for (const log of logs) {
      if (!log.startsWith(PROGRAM_LOG_PREFIX)) continue;
      const base64Data = log.slice(PROGRAM_LOG_PREFIX.length);
      const buffer = Buffer.from(base64Data, "base64");
  ```

  ```typescript
    error?: SimulationError;
    heliusEnhanced: boolean;
  }

  export type RiskFlag =
    | "LARGE_OUTFLOW"       // > spending cap in single tx

    | "FULL_DRAIN"          // vault balance → 0
    | "MULTI_OUTPUT"        // funds scattered to 3+ wallets
    | "SIZE_OVERFLOW";      // tx exceeds 1232B

  export interface BalanceDelta {
    address: Address;
    mint: Address;           // SOL native or SPL token mint
    before: bigint;
    after: bigint;
    delta: bigint;           // negative = outflow
  }
  ```
- **Key functions**:
  - `simulateBeforeSend(rpc, transaction)` → `SimulationResult`
  - `extractBalanceDeltas(simulation, knownAccounts)` → `BalanceDelta[]`
  - `detectDrainAttempt(deltas, vaultAddress, allowedRecipients)` → `{ safe: boolean, flags: RiskFlag[], details: string }`
- **Helius detection**: Check if RPC URL contains "helius" → use enhanced simulation fields (`preBalances`, `postBalances`, `preTokenBalances`, `postTokenBalances`).
- **Fail-closed**: Simulation failure = transaction blocked. Never sign what can't be simulated.

- **Simulation timeout**: `simulateBeforeSend()` wraps Kit's `rpc.simulateTransaction()` with `AbortSignal.timeout(timeoutMs)`. Default: 3,000ms. Configurable via `PhalnxKitClientOptions.simulationTimeoutMs`. On timeout: throw `AgentError(SIMULATION_TIMEOUT, recovery: ["Retry with shorter route", "Skip simulation with opt-out"])`.
- **Non-Helius fallback**: When RPC URL does NOT contain "helius":
  - Standard `simulateTransaction()` still provides: `err`, `logs`, `unitsConsumed`
  - Missing: `preTokenBalances`, `postTokenBalances` → balance deltas unavailable
  - Fallback behavior: set `heliusEnhanced: false`, return empty `balanceDeltas[]`, skip drain detection (can't detect without balance data)
  - Log warning: `"Non-Helius RPC: balance delta analysis unavailable. Drain detection disabled."`
- **CU auto-adjustment**: After successful simulation, read `simulation.computeUnits` and update the ComputeBudget instruction:
  ```typescript
  const adjustedCU = Math.ceil(simulation.computeUnits * 1.1); // 10% headroom
  // Re-compose transaction with adjusted CU if significantly different from estimate
  if (Math.abs(adjustedCU - estimatedCU) > estimatedCU * 0.2) {
    txMessage = await recomposeWithCU(txMessage, adjustedCU);
  }
  ```
- **Simulation opt-out**: `PhalnxKitClientOptions.skipSimulation?: boolean` — when `true`, skip `simulateBeforeSend()` entirely. Intended for testing environments ONLY. When enabled, log warning: `"Simulation disabled — drain detection and balance validation bypassed"`. NOT exposed in MCP tools (agents should never skip simulation).
- **Simulation divergence handling**: When simulation succeeds but `sendAndConfirm` fails:
  1. Parse the on-chain error → `toAgentError()`
  2. If error is `BlockhashNotFound` or `AlreadyProcessed` → retry with fresh blockhash (max 2 retries)
  3. If error is any other → return `AgentError` with `context.simulationPassed: true` flag so agent knows the simulation was OK but execution diverged
  4. Include `recovery_actions: ["Transaction may have landed — check vault balance before retrying"]`
- **Simulation for vault management ops**: `createVault`, `registerAgent`, `updatePermissions` etc. are simulated for CU estimation only (no drain detection — they don't move tokens). `simulateBeforeSend()` detects vault management by checking: no DeFi program IDs in instructions. Skip drain detection, return CU only.

#### Step 2.8: Build `intent-engine.ts` — the main agent-facing facade

- **File**: `sdk/kit/src/intent-engine.ts`
- **Contents**: Kit-native IntentEngine wiring all pieces together.
  ```typescript
  export class IntentEngine {
    constructor(private client: PhalnxKitClient) {}

    async execute(intent: IntentAction, vault: Address): Promise<ExecuteResult> {
      // 1. Validate intent input (anti-hallucination)
      const validation = validateIntentInput(intent);
      if (!validation.valid) throw validation.errors[0];

      // 2. Resolve protocol tier
      const resolution = resolveProtocol(intent.protocol ?? inferProtocol(intent));

      // 3. Resolve tokens (symbol → Address + decimals)
      const inputToken = await resolveTokenAsync(intent.params.inputToken, this.client.rpc);
      const outputToken = intent.params.outputToken
        ? await resolveTokenAsync(intent.params.outputToken, this.client.rpc)
        : undefined;

      // 4. Precheck (mandatory — permission, cap, allowlist, constraints)
      const precheck = await this.precheck(intent, vault, inputToken);
      if (!precheck.allowed) throw toAgentError(precheck.reason, precheck.context);

      // 5. Build instructions via protocol handler
      const handler = globalProtocolRegistry.getHandler(resolution.protocolId);
      if (!handler && resolution.tier !== ProtocolTier.T4_PASSTHROUGH) {
        throw protocolEscalationError(resolution.tier, resolution.protocolId);
      }
      const composed = handler
        ? await handler.compose(ctx, resolvedAction, resolvedParams)
        : { instructions: intent.params.rawInstructions };  // T4 passthrough

      // 6. Verify adapter output (safety gate)
      const verified = verifyAdapterOutput(
        composed.instructions,
        handler?.metadata.programIds ?? [],
        [vault],
      );
      if (!verified.valid) throw new Error(`Adapter violation: ${verified.violations.join(", ")}`);

      // 7. Resolve all PDA accounts
      const accounts = resolveAccounts({ vault, agent, tokenMint: inputToken.mint, ... });

      // 8. Build validate_and_authorize + finalize_session instructions
      const validateIx = getValidateAndAuthorizeInstruction({ ...accounts, ...params });
      const finalizeIx = getFinalizeSessionInstruction({ ...accounts });

      // 9. Compose sandwich via pipe()
      const txMessage = await composePhalnxTransaction({
        rpc: this.client.rpc,
        feePayer: this.client.feePayer,
        validateAndAuthorizeIx: validateIx,
        defiInstructions: composed.instructions,
        finalizeSessionIx: finalizeIx,
        computeUnits: estimateComposedCU(composed.instructions),
        priorityFeeMicroLamports: await this.client.estimatePriorityFee(),
      });

      // 10. Simulate pre-sign (fail-closed)
      const simulation = await simulateBeforeSend(this.client.rpc, txMessage);
      if (!simulation.success) throw toAgentError(simulation.error, { simulation });
      const drainCheck = detectDrainAttempt(simulation.balanceDeltas, vault, allowedRecipients);
      if (!drainCheck.safe) throw toAgentError({ flags: drainCheck.flags }, { simulation });

      // 11. Sign and send
      const signedTx = await signTransactionMessageWithSigners(txMessage);
      const sendAndConfirm = sendAndConfirmTransactionFactory({
        rpc: this.client.rpc,
        rpcSubscriptions: this.client.rpcSubscriptions,
      });
      await sendAndConfirm(signedTx, { commitment: "confirmed" });

      // 12. Return result with signature + events
      const signature = getSignatureFromTransaction(signedTx);
      const events = parsePhalnxEvents(simulation.logs);

      return {
        signature,
        computeUnits: simulation.computeUnits,
        balanceDeltas: simulation.balanceDeltas,
        events,
      };
    }
  }
  ```

**Agent manipulation defense Layer 1**: Steps 10-11 (simulate + drain detect) constitute Layer 1. See Step 5.10 (VelocityTracker) for Layer 2, Step 5.11 (intent-drift) for Layer 3.

- **File**: `sdk/kit/src/integrations/codama-handler.ts`
- **Contents**:
  ```typescript
  export class CodamaHandler implements ProtocolHandler {
    readonly metadata: ProtocolHandlerMetadata;

    constructor(
      protocolId: string,
      displayName: string,
      programIds: Address[],
      private idl: AnchorIdl,
      private actionMap: Map<string, { instructionName: string; actionType: Record<string, Record<string, never>>; isSpending: boolean }>,
    ) {
      this.metadata = { protocolId, displayName, programIds, supportedActions: this.buildActionDescriptors() };
    }

    async compose(ctx: ProtocolContext, action: string, params: Record<string, unknown>): Promise<ProtocolComposeResult> {
      const mapping = this.actionMap.get(action);
      if (!mapping) throw new Error(`Unknown action: ${action}`);

      const ixDef = this.idl.instructions.find(i => i.name === mapping.instructionName);
      if (!ixDef) throw new Error(`Instruction ${mapping.instructionName} not found in IDL`);

      const data = serializeInstruction(ixDef, params);
      const accounts = await resolveIdlAccounts(ixDef, ctx, params, this.idl);

      return {
        instructions: [{
          programAddress: this.metadata.programIds[0],
          accounts,
          data,
        }],
      };
    }
  }
  ```

### Phase 10: Policy-in-TEE — Level B Custody Fusion (P3 — 2-4 weeks)

**Origin**: Level 4 architectural analysis (2026-03-12). Council-approved phased custody fusion: Level A (wrapper, Phase 8.4a-d) → **Level B (policy-in-TEE, this phase)** → Level C (on-chain-gated signing, future/monitor).

**Goal**: Move policy evaluation INSIDE the TEE custody boundary so a compromised host cannot bypass guardrails. Currently, Shield runs on the host and Turnkey signs in the TEE — a compromised host can skip Shield and call Turnkey directly. Level B eliminates this gap.

**Prerequisite**: Phase 5 (VelocityTracker + Intent-Drift), Phase 8.4a-d (CustodyAdapters + PhalnxSigner)

#### Step 10.1: Standalone Policy Evaluator

- **File**: `sdk/kit/src/policy-evaluator.ts` (~300 lines)
- **Purpose**: Extract Shield's policy evaluation logic (instruction allowlist, velocity, intent-drift, spend limits) into a pure, dependency-free module. No RPC calls, no SDK imports — only takes serialized transaction bytes + policy config, returns allow/deny.
- **Why standalone**: TEE runtimes (Nitro, SGX) have restricted environments. The evaluator must run without Node.js SDK dependencies.
- **Interface**:
  ```typescript
  interface PolicyEvaluatorInput {
    transactionBytes: Uint8Array;
    policyConfig: SerializedPolicyConfig;
    velocityState: SerializedVelocityState;
    intentContext?: SerializedIntentContext;
  }

  interface PolicyEvaluatorResult {
    allowed: boolean;
    violations: PolicyViolation[];
    velocityStateAfter: SerializedVelocityState;
  }

  function evaluatePolicy(input: PolicyEvaluatorInput): PolicyEvaluatorResult;
  ```
- **Test**: `sdk/kit/tests/policy-evaluator.test.ts` (20 tests)
  - Verify: identical decisions to Shield for all existing test scenarios
  - Verify: zero external dependencies (import graph check)
  - Verify: deterministic output (same input → same output, no randomness)

WIP commit: `[WIP Phase 10.1] refactor: extract standalone policy evaluator from Shield`

#### Step 10.2: Turnkey Custom Policy Integration

- **File**: `sdk/custody/turnkey/src/tee-policy.ts` (~200 lines)
- **Purpose**: Integrate the standalone policy evaluator with Turnkey's policy engine API. Turnkey supports custom signing policies that run inside the Nitro enclave alongside the key. This step uploads the policy evaluator as a Turnkey policy.
- **Turnkey policy mechanism**: Turnkey's `createPolicy` API accepts policy rules that gate signing. Level B uses this to ensure every sign request passes `evaluatePolicy()` inside the enclave before the key is used.
- **Configuration**: Policy config (spend limits, protocol allowlist, velocity thresholds) is set at policy creation time and updatable via Turnkey's policy update API (which itself requires organization-level auth).
- **Fallback**: If Turnkey policy API is unavailable or doesn't support custom evaluators at integration time, fall back to Turnkey's native policy rules (amount limits, destination allowlist) as a partial Level B.

WIP commit: `[WIP Phase 10.2] feat: Turnkey TEE policy integration (Level B custody fusion)`

#### Step 10.3: Attestation Extension — Policy Code Hash

- **File**: Update `sdk/custody/turnkey/src/kit-adapter.ts`
- **Purpose**: Extend `attestation()` to return the policy evaluator's code hash alongside the existing Nitro enclave attestation. This lets verifiers confirm that the correct policy logic is running inside the TEE.
- **AttestationResult extension**:
  ```typescript
  interface AttestationResult {
    // existing fields...
    policyCodeHash?: string;   // SHA-256 of the policy evaluator deployed in TEE
    policyVersion?: string;    // Semantic version of the policy evaluator
    policyConfig?: string;     // Hash of the active policy config (limits, allowlist)
  }
  ```
- **Verification**: Client-side can compare `policyCodeHash` against a known-good hash published in the SDK release.

WIP commit: `[WIP Phase 10.3] feat: attestation with policy code hash verification`

#### Step 10.4: Compromised-Host Bypass Test

- **File**: `sdk/custody/turnkey/tests/tee-policy-bypass.test.ts` (~100 lines)
- **Purpose**: Integration test proving that a compromised host CANNOT sign a policy-violating transaction.
- **Test scenarios**:
  1. Host sends over-limit transaction directly to Turnkey → Turnkey rejects (policy evaluator inside TEE blocks it)
  2. Host sends allowlisted transaction → Turnkey signs (policy passes)
  3. Host attempts to update policy config without org auth → rejected

## Part 14: Scope Exclusions (What We Are NOT Building)

1. **Actions/Blinks Bridge** — Stress tests proved it's Jupiter-only. Already covered by T1_API.
2. **Community handler marketplace** — Premature before volume. Build after 10+ handlers exist.
3. **DeFi Action Schema standard** — Publish after adoption validates the taxonomy.
4. **On-chain constraint authoring UI** — Constraint templates (Phase 4.5) reduce the need. Full UI is dashboard work. _Note: SDK backend (ConstraintBuilder + protocol descriptors) is now implemented — see `sdk/kit/src/constraints/`._
5. **On-chain program changes** — The Anchor Rust program is unchanged. Only the TypeScript client layer changes.
6. **Anchor Kit support** — We don't wait for Anchor to adopt Kit. Codama replaces Anchor's TS layer.
7. **LiteSVM Kit-native test harness** — On-chain tests stay on web3.js (test code, not prod code). Kit SDK tests use mock RPC or devnet/Surfpool via Kit's `createSolanaRpc()`.
8. **Full-auto protocol onboarding** — ActionType mapping (step 3 of T3 flow) requires human judgment. AI-assisted classification is a stretch goal, not a requirement.
9. **`freeze_and_pause_all` batch instruction** — Emergency response audit suggested a single instruction to freeze vault + pause all agents atomically. Risk: adds new on-chain instruction surface area. Current approach (freeze vault → agents auto-blocked) is sufficient. Revisit if user testing reveals freeze latency issues.
10. **Token-2022 support** — Token extensions (transfer hooks, confidential transfers, metadata) require significant on-chain changes to `is_stablecoin_mint()`, ATA derivation, and fee calculation. Out of scope until Token-2022 stablecoins gain market share.
11. **x402 v2 enhancements** — Content-type negotiation, streaming payment support, retry with different payment methods. Current `shieldedFetch()` covers the core use case. Revisit after x402 protocol stabilizes.
12. **`@phalnx/constraint-templates` npm package** — Phase 4.5 generates constraint configs in the SDK. A standalone npm package for template distribution is premature. Build after 10+ community-created templates exist. _Note: Protocol descriptors (`FlashTradeDescriptor`, `KaminoDescriptor`) now serve as built-in templates._
13. **Guardian delegation model** — Multi-signature recovery with designated guardians who can freeze/unfreeze without owner key. Requires on-chain changes (new PDA type, new instructions). Complex security implications. Revisit post-mainnet.
14. **Time-boxed pause (auto-resume)** — Agent pause with automatic resume after N slots. Requires on-chain timer logic. Current manual unpause is sufficient and safer (forces explicit review).

### Future Enhancements (from roadmap audit — TradFi patterns worth adapting)

These SDK-layer enhancements require zero on-chain changes. Recommended for Phase 2.5 or Phase 5.5:

15. **Pre-trade what-if analysis** (from TradFi OMS pattern) — Add `whatIfAnalysis()` to IntentEngine pipeline between precheck and compose. Computes projected vault state (balances, exposure, concentration) and evaluates against configurable policy thresholds. Catches concentration risk and minimum reserve violations before spending compute on transaction composition. Pattern: FIX Protocol pre-trade compliance.
16. **Failure state machine + circuit breaker** (from SWIFT exception workflows) — Add structured failure tracking to IntentStorage: `{ intentId, attempts: [{ timestamp, errorCode, recoveryAction }], state: 'pending' | 'failed' | 'retrying' | 'resolved' }`. Add per-protocol circuit breaker (Resilience4j pattern: closed → open → half-open) that tracks success/failure rates and temporarily degrades failing handlers.
17. **Intent-to-execution reconciliation** (from MiFID II record-keeping) — The correlation portion is now Step 5.11 (`checkIntentDrift`). What remains: `correlationId` in on-chain events (via memo instruction in composed tx), pluggable IntentStorage backend, `reconcile()` method comparing simulation vs actual results post-confirmation.
